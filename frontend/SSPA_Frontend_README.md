# SSPA Frontend

## Project Overview

The SSPA Frontend is the React/Vite user interface for the **Smart Success Predictive Analytics (SSPA)** system. It provides role-based dashboards for Students, Lecturers, Academic Advisors, and Admin users.

The frontend connects to the FastAPI backend API Gateway and displays student risk predictions, alerts, interventions, course summaries, and reporting/export features.

The main goal of the frontend is to support early identification of students who may be at risk of not qualifying or performing poorly, and to help lecturers and advisors manage academic interventions.

## Main Frontend Features

### 1. Role-Based Access

The system supports the following roles:

- STUDENT
- LECTURER
- ADVISOR
- ADMIN

Each role has its own dashboard view and access level.

Demo role passwords:

```text
STUDENT  -> Student@123
LECTURER -> Lecturer@123
ADVISOR  -> Advisor@123
ADMIN    -> Admin@123
```

The frontend sends login details to the backend endpoint:

```text
POST /me/simulate-login
```

After successful login, the frontend stores:

```text
sspa_access_token
sspa_session
sspa_role
sspa_token_expires_at
```

The access token is then sent with protected API requests using:

```text
Authorization: Bearer <token>
```

## Dashboard Modules

### Lecturer Dashboard

The Lecturer Dashboard allows a lecturer to:

- View courses assigned to them
- View students in their courses
- Search students across all courses taught by that lecturer
- View risk heatmaps
- View On Track, Moderate Risk, and High Risk student counts
- Click summary cards to filter students accurately
- Create interventions only for at-risk students
- Start, complete, or escalate intervention cases
- View alert badges
- Export reports

Important lecturer rules:

- A lecturer can only search students in courses assigned to that lecturer.
- Interventions can only be created for Moderate Risk or High Risk students.
- On Track students should not have intervention cases created unnecessarily.

#### Lecturer Intervention Pipeline

Interventions follow this pipeline:

```text
TRIGGERED -> IN_PROGRESS -> COMPLETED
```

The lecturer can:

- Click **Start** to move a triggered intervention to `IN_PROGRESS`
- Click **Complete** to move an in-progress intervention to `COMPLETED`
- Click **Escalate** to move a case to `ESCALATED`

The dashboard displays counts for:

- Triggered
- In Progress
- Completed
- Escalated

When a case is escalated, an advisor should receive an alert.

---

### Student Dashboard

The Student Dashboard allows a student to:

- View their own risk status
- View their academic information
- View alerts and notifications
- View support/intervention information where applicable

Important student rule:

- A student should only see their own academic records and alerts.

---

### Advisor Dashboard

The Advisor Dashboard allows academic advisors to:

- View escalated intervention cases
- View advisor alerts
- Review students needing additional academic support
- Track escalated cases from lecturers
- Export reports

Important advisor rule:

- Advisors should only see cases that are escalated or assigned to them.

---

### Admin Dashboard

The Admin Dashboard is used for system-level monitoring, including backend modules and feature store status.

Important admin rule:

- Data export buttons must not be shown on the Admin Dashboard.
- Export buttons are visible only on Lecturer and Advisor dashboards.

## Alerts and Notifications

The notification module shows unread alert counts on dashboard sidebars.

Alert badges are displayed for:

- Lecturer alerts
- Student alerts
- Advisor alerts


Before automation, alerts were mostly fetched as existing notifications. After automation, prediction results can trigger alerts automatically when students are identified as Moderate Risk or High Risk.

Advisor alerts must also be created when a lecturer escalates an intervention case.

## Predictive Analytics Display

The frontend displays predictions generated by the backend Random Forest classifier.

The prediction output includes:

- Student number
- Course code
- Risk probability
- Risk level
- Recommended action or alert message

Risk categories:

```text
ON_TRACK
MODERATE
HIGH
```

Example:

```
Student sXXXX has been identified as high risk in course XXXX.
Please review the student profile and create an intervention if needed.
```

The percentage is a risk probability, not a student mark.

## Data Export / Reporting

Only the Lecturer and Advisor dashboards must show visible data export buttons.

Required export buttons:

- Export At-Risk Students
- Export Course Summary
- Export Interventions

The frontend service file responsible for exports is:

```text
src/services/reportExportApi.js
```

Expected backend endpoint example:

```text
GET /reports/lecturer/export.csv
```

The export service sends the bearer token:

```text
Authorization: Bearer <token>
```

If exports return `404 Not Found`, ensure the backend reports router is registered in:

```text
api-gateway/app/main.py
```

Example backend router registration:

```python
from app.routers import reports

app.include_router(reports.router)
```

## Authentication Flow

1. User selects role.
2. User enters role password.
3. User enters identifier such as:
   - Student number
   - Lecturer number
   - Advisor number
4. Frontend calls:

```text
POST /me/simulate-login
```

5. Backend returns:

```text
access_token
session
token_type
expires_at
```

6. Frontend stores the token and session in localStorage.
7. Protected API calls include:

```text
Authorization: Bearer <access_token>
```

## Common Troubleshooting

### Blank screen after login

Run:

```powershell
cd C:\SAKAILMSPLUGIN\frontend
npm run build
```

If the build passes, clear browser storage:

```javascript
localStorage.clear();
sessionStorage.clear();
location.href = "/";
```

### Export button says Not Found

Check that the backend reports router is registered in:

```text
api-gateway/app/main.py
```

Also confirm the route exists in:

```text
http://localhost:8000/docs
```

### `reportExportApi.js: url is not defined`

This happens when unused code exists outside the `downloadReportCsv` function.

Make sure there is no code after the closing brace of:

```javascript
export async function downloadReportCsv(...) {
  ...
}
```

### Unauthorized admin or dashboard request

Make sure the frontend sends:

```text
Authorization: Bearer <token>
```

Also logout and login again to refresh the token.

### Vite import error

If Vite says a service file cannot be resolved, confirm the file exists under:

```text
src/services/
```

Example:

```text
src/services/reportExportApi.js
```

## Security Notes

This frontend uses static demo passwords for project demonstration. In production, passwords must not be stored in frontend code.

Production security should include:

- Backend-controlled authentication
- Hashed passwords stored securely in the backend
- HTTPS
- Role-based backend authorization
- Secure environment variables
- No sensitive credentials in source code

Even though the frontend controls navigation, the backend must still enforce role checks. Frontend role switching alone is not enough security.
